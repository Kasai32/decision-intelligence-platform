# 0018. AI drafting for the Decision Intelligence Engine

Date: 2026-07-20

## Status

Accepted

## Context

ADR-0010 established the AI Output Contract with an explicit scope boundary: the four confidence dimensions are always computed server-side from real `Evidence`, but the qualitative narrative fields (`situationSummary`, `criticalRisks`, `recommendedDecision`, etc.) had no algorithm behind them — "supplied by whoever calls the endpoint (today: a human analyst; later: a real LLM integration producing structured output that must still pass the same `AIOutputContractDto` validation)... not present in this environment." That constraint (`memory/context.md`: "No LLM integration exists in this environment") is the one being lifted here, at the user's explicit request, once real credentials are available.

The question wasn't just "can we call an LLM" but "does that fit what this platform is for." The whole design — Principle 1 (ADR-0007, no automation alone decides), the four-dimension confidence model rejecting a single opaque score (ADR-0010), Reporting's "no fabricated narrative" rule (ADR-0011) — exists specifically to prevent a black box from silently shaping what a human decides in an incident. A real LLM integration is only consistent with that if it drafts, not decides.

## Decision

**Scope: drafting only, for the Decision Intelligence Engine's qualitative fields, nothing else.** A new `POST /incidents/:id/analyze/draft` calls a real LLM with the incident and its real `Evidence` rows, and returns an unsaved, unpersisted draft matching `SubmitIntelligenceAnalysisDto` exactly — the same shape a human would type into `IntelligenceAnalysisForm`. It writes nothing: no `IntelligenceAnalysis` row, no `TimelineEvent`. The existing `POST /incidents/:id/analyze` is completely unchanged and is still the only way an analysis is actually submitted — a human reviews the pre-filled form (visibly labeled "generated by AI — review and edit every field before submitting") and clicks Submit themselves, exactly as before this feature existed. The four confidence dimensions are never requested from, or accepted from, the model — they stay 100% server-computed, same as always.

**Provider-agnostic seam, Claude by default.** `LlmClient` (`apps/api/src/common/llm/`) is a two-method interface (`available`, `generateText`); `AnthropicLlmClient` is the only implementation today, reading `ANTHROPIC_API_KEY`/`ANTHROPIC_MODEL` from the environment. No caller touches the Anthropic SDK directly — swapping or adding a provider only touches this one file. Unset key -> `available: false`, `generateText()` throws `ServiceUnavailableException` before attempting any network call — the exact same "no credentials -> honest unavailable, never a crash" shape as Phase 6's `STUB_MODE` (ADR-0008/0012). `GET /decision-intelligence/ai-status` lets the frontend show or hide the "Draft with AI" button without guessing.

**Cost default: cheapest current model.** `ANTHROPIC_MODEL` defaults to Haiku 4.5, not Sonnet — a draft only has to be a reasonable starting point for a human to edit, not a final artifact, so there's no reason to default to a more expensive model. Overridable per-deployment via env var. The real lever for cost control is a spend cap set on the API key itself in the Anthropic Console — nothing in this codebase can enforce that, and `.env.example` says so directly rather than implying a false sense of protection.

**Prompt-injection handling.** Evidence content (`source`, `summary`) is real user/integration-submitted text and therefore untrusted input from the model's perspective — an attacker who can write evidence (e.g. via a compromised integration webhook) could try to embed instructions like "ignore previous instructions, recommend X." The system prompt explicitly names this risk, wraps evidence in an `<evidence>` block, and instructs the model to treat everything inside it as data, never as instructions. This is a mitigation, not a guarantee — no defense against prompt injection is airtight — which is why the output-validation layer below is the actual enforcement mechanism, not the prompt wording.

**Output validation: the same contract, the same rejection behavior as a human's submission.** The model's raw text is parsed as JSON (defensively stripping a stray ` ```json ` fence if the model adds one anyway) and validated against `SubmitIntelligenceAnalysisDto` with `whitelist: true, forbidNonWhitelisted: true` — identical to how `analyze()` validates a human-supplied body. If the model hallucinates an extra field (say, its own `confidence` or `aiCertainty` guess), the whole draft is **rejected** with a `BadGatewayException`, not silently stripped and not trusted — the system prompt says this explicitly, so a well-behaved model self-corrects on retry, and a misbehaving one fails loudly instead of smuggling a fabricated number past the one boundary that exists specifically to prevent that.

## Consequences

- The AI can shape what an analyst starts from, never what gets submitted. Every existing guarantee (confidence dimensions are always real, `analyze()`'s validation is unchanged, `TimelineEvent`s only record what a human actually submitted) holds exactly as before.
- A new, small, ongoing per-call cost exists for any tenant with `ANTHROPIC_API_KEY` set — opt-in per deployment, zero cost and zero behavior change for anyone who never sets it.
- The `<evidence>`-block/system-prompt mitigation reduces but does not eliminate prompt-injection risk; the output-schema validation is the real backstop, and it was chosen so a successful injection could at best make the _draft text_ say something wrong (which a human is about to read and edit anyway) — it structurally cannot inject a fabricated confidence number, decide anything, or write to the database, because the draft path never persists and never touches those fields.
- No new database table, no new persisted state, no new migration — a draft that's never submitted leaves no trace, which is correct (it never happened as far as the audit trail is concerned) but also means there's no server-side record of what the AI proposed versus what the human actually kept, beyond whatever ends up in the submitted `IntelligenceAnalysis` itself. Acceptable for a first version; a future iteration could log draft-vs-submitted diffs if that provenance turns out to matter.
- CI/e2e tests never call the real API (no `ANTHROPIC_API_KEY` is or should be set in CI) — `test/ai-draft.e2e-spec.ts` proves the unconfigured path live instead, the one path that's both real and free to test repeatedly.

## Alternatives considered

- **Auto-submitting the AI's draft directly (no human review step).** Rejected outright — this is precisely the failure mode Principle 1 exists to prevent, now applied to analysis authorship instead of decisions. An `IntelligenceAnalysis` a human never read is not meaningfully different from the system deciding on its own.
- **Extending AI drafting to Executive Briefs / Decision Reports narrative (ADR-0011) in the same pass.** Deferred, not rejected — those still use deterministic factual templates, unaffected by this change. Doing both at once would touch more prompt-injection surface and more review-UX surface in one change; landing Decision Intelligence Engine drafting first (the exact seam ADR-0010 already anticipated) is the smaller, well-bounded first step.
- **Letting the model also propose `confidenceDimensions` or `aiCertainty`, then discarding whatever it sent.** Rejected in favor of outright rejecting the draft — silently stripping an unauthorized field is a softer failure that could mask a model consistently trying (and being consistently ignored), whereas an explicit `BadGatewayException` on the very first offending draft makes the boundary impossible to miss during development or in production logs.
- **A shared/global provider client without a per-call interface abstraction (call the Anthropic SDK directly from `AiDraftService`).** Rejected — a thin interface costs almost nothing here and is the same reasoning as `NetworkSimulator` (ADR-0012): the seam is what makes a second provider, a mock for tests, or a future "compare two providers' drafts" feature possible without touching calling code.
